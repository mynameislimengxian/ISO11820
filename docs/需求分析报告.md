## 1. 项目概述

### 1.1 项目背景

在建筑防火安全领域，建筑材料的不燃性（Non-combustibility）是评估材料防火性能的核心指标。国际标准化组织制定的 ISO 11820:2020《建筑材料不燃性试验方法》规定了严格的试验流程：将建筑材料样品放入加热炉，加热至 750°C，持续记录 60 分钟的温度数据，通过分析炉温、样品表面温度、样品中心温度的变化以及样品失重率，判定材料是否满足"不燃"标准。

然而，传统的建筑材料不燃性试验存在以下痛点：

| 痛点 | 描述 |
|------|------|
| 人工依赖度高 | 从升温、稳定判断、数据记录到报告生成，全程依赖操作人员手动操作和主观判断 |
| 过程可控性差 | 温度稳定性的判断依赖人工经验，缺乏客观、科学的判定标准 |
| 数据一致性低 | 人工记录方式容易出现遗漏、错误，不同操作员之间的数据质量参差不齐 |
| 效率低下 | 单次试验需持续 60 分钟以上，全程人工值守，人力成本高 |
| 标准符合性难保证 | ISO 11820 标准对温度控制精度、数据记录频率、结果计算均有严格要求，人工操作难以完全满足 |

本项目正是基于上述背景，参考《建筑材料不燃性实验自动化测控系统研究与实现》（罗凌岚，2026）毕业设计成果，面向软件工程实践课程考核，由四人团队协作，从零开发一套仿真模式下的建筑材料不燃性试验系统。该系统无需真实硬件，通过软件仿真模拟整个试验流程，涵盖登录、试验创建、自动升温、温度稳定判定、数据记录、试验现象记录、报告导出等完整功能。每位成员借助 AI 大模型辅助完成各自的文档和代码模块。

### 1.3 项目目标

**总体目标**：构建一套严格遵循 ISO 11820:2020 标准流程、以仿真引擎替代真实硬件、实现建筑材料不燃性试验全流程自动化管理的桌面应用系统。

**具体目标**：

| 编号 | 目标 | 衡量指标 |
|------|------|---------|
| G1 | 实现完整的试验流程自动化 | 从登录到报告导出，全流程无需人工干预 |
| G2 | 仿真引擎准确模拟 5 通道温度变化 | 炉温稳定在 750°C±5°C，升温/稳定/记录/降温阶段行为符合物理规律 |
| G3 | 科学判定温度稳定性 | 基于 MathNet.Numerics 线性回归算法的温漂计算，替代人工主观判断 |
| G4 | 生成标准格式报告 | 支持 CSV、Excel（含图表）、PDF 三种格式导出 |
| G5 | 数据持久化与可追溯 | 所有试验数据存入 SQLite 数据库和 CSV 文件，支持历史查询 |

### 1.4 项目范围

**包含的功能**：

1. 角色选择与密码登录（管理员 / 试验员）
2. 试验信息创建（样品信息、环境参数、操作员、时长模式）
3. 仿真温度引擎（5 通道温度数据，每 800ms 更新一次）
4. 五阶段试验状态机（Idle → Preparing → Ready → Recording → Complete）
5. 实时温度数值显示与 OxyPlot 曲线图
6. 系统消息日志（时间戳 + 事件内容）
7. 按钮状态按状态机自动控制
8. 试验现象记录（火焰观察、试验后质量、失重率计算）
9. CSV / Excel / PDF 三种格式数据导出
10. 历史试验记录查询（按日期、样品编号、操作员筛选）
11. 设备校准记录管理

**不包含的功能**：

| 功能 | 排除原因 |
|------|---------|
| 真实串口 / Modbus 通信 | 课程实训环境无硬件，由仿真引擎替代 |
| 真实 PID 控制算法 | 仿真模式下不需要，由仿真算法替代 |
| 摄像头火焰检测 | 需要摄像头硬件，Demo 中跳过 |
| 多炉同时控制 | 简化设计，只做单炉 |
| ISO 标准完整合规判定 | 教学简化，仅计算温升和失重率 |
| 网络 / 云端同步 | 纯本地运行 |
| 复杂权限管理（RBAC） | 简单角色区分（管理员/试验员）即可 |

#### 2.1.2 技术栈对比分析

| 对比维度 | 本方案（WinForms + SQLite） | 替代方案 1（WPF + SQL Server） | 替代方案 2（Web + MySQL） |
|---------|---------------------------|------------------------------|--------------------------|
| 开发难度 | 低：WinForms 拖拽式开发 | 中：WPF 需学习 XAML/MVVM | 中高：需前后端分离 |
| 部署复杂度 | 极低：单文件发布即可运行 | 中：需安装 .NET Runtime | 高：需部署 Web 服务器 |
| 硬件依赖 | 无（纯仿真） | 无（纯仿真） | 需网络环境 |
| 环境要求 | Windows 10/11 | Windows 10/11 | 跨平台 |
| 学习成本 | 低：课程已教授 | 中：需额外学习 | 高：需掌握多门技术 |
| 适合场景 | 单机桌面应用 | 复杂桌面应用 | 多用户联网应用 |

**结论**：本方案选择 WinForms + SQLite 是对课程教学场景的最优解——开发效率高、部署简单、学习成本低，完全满足项目需求。

#### 2.1.3 核心技术难点及解决方案

| 技术难点 | 难点分析 | 解决方案 | 可行性 |
|---------|---------|---------|--------|
| 温度仿真算法 | 需模拟 5 通道温度在不同阶段的物理行为，要求逼真且可配置 | 分阶段仿真模型：升温阶段线性增长+噪声，稳定阶段钳位+噪声，记录阶段指数逼近+噪声 | 已在前置研究中验证，算法简单可靠 |
| 状态机设计 | 5 个状态之间有多种合法与非法转移路径，需严格管控 | 使用 lock 保护状态转移，每个操作前检查当前状态合法性，返回布尔值指示成功/失败 | 经典设计模式，实现风险低 |
| 实时曲线更新 | 每 800ms 更新一次 4 条曲线，需保证 UI 不卡顿 | OxyPlot 支持增量数据追加，定时器在后台线程运行，通过 Invoke 安全更新 UI | OxyPlot 原生支持动态数据 |
| 跨线程数据广播 | 后台线程生成数据，UI 线程消费数据，需避免跨线程异常 | 使用 C# 事件机制，在 DataBroadcast 事件回调中使用 Control.Invoke() 切回 UI 线程 | 标准 WinForms 跨线程操作模式 |
| Excel 图表嵌入 | 需在 Excel 报告中嵌入温度曲线图 | EPPlus 7.x 原生支持图表创建和数据绑定 | EPPlus 官方文档完整 |
| PDF 中文支持 | PDF 生成需要正确渲染中文 | PDFsharp-MigraDoc 6.x 支持 Unicode，配置中文字体即可 | 已有成熟方案 |

#### 2.1.4 技术可行性结论

技术方案完全可行。所选技术栈均为成熟稳定的主流方案，核心技术难点均有明确的解决路径。项目无需引入未经验证的新技术，开发风险低。

## 3. 业务需求分析

### 3.1 业务流程描述

完整的建筑材料不燃性试验业务流程分为以下步骤：

**步骤 1：登录系统。** 用户启动程序，选择角色（管理员或试验员），输入密码，系统验证通过后进入主界面。

**步骤 2：新建试验。** 用户点击"新建试验"，在弹出的窗口中填写环境温度、环境湿度、样品编号、样品名称、规格、直径、高度、试验前质量、操作员、试验时长模式，设备信息自动带入。点击"创建试验"保存到数据库。

**步骤 3：开始升温。** 用户点击"开始升温"按钮，系统状态从 Idle 变为 Preparing。仿真引擎每 800ms 更新一次温度，炉温从室温开始向 750°C 上升，实时曲线同步更新。

**步骤 4：温度稳定。** 系统自动检测：当炉温进入 745~755°C 范围且持续稳定（稳定计数器大于 3 次 tick，约 3.2 秒），状态自动从 Preparing 变为 Ready，系统提示"温度已稳定，可以开始记录"。

**步骤 5：开始记录。** 用户点击"开始记录"按钮，状态从 Ready 变为 Recording。计时器开始计数，系统每秒记录一行温度数据（5 通道），同时写入 CSV 文件。

**步骤 6：停止记录。** 满足以下任一条件时进入 Complete 状态：（a）标准 60 分钟模式到达 3600 秒自动终止；（b）固定时长模式到达设定秒数；（c）用户手动点击"停止记录"。

**步骤 7：填写现象记录。** 用户点击"试验记录"打开现象记录窗口，填写是否出现持续火焰、火焰发生时刻和持续时间（如有）、试验后质量（必填）、备注。系统自动计算：失重量 = 试验前质量 - 试验后质量，失重率 = 失重量 / 试验前质量 × 100%，各通道温升。保存后 flag 设为 "10000000"。

**步骤 8：导出报告。** CSV 文件在试验完成时自动生成；PDF 在保存试验记录后自动生成；Excel 支持手动触发导出。用户可在"记录查询"Tab 中查看和导出历史试验记录。

### 5.3 数据流

系统存在三条主要数据流：

**数据流 1：温度数据推送。** 仿真引擎（后台线程）每 800ms 通过 DataBroadcast 事件向 UI 层推送温度数据（TF1、TF2、TS、TC、TCal）、当前状态、已记录秒数和系统消息。UI 层在 Invoke 回调中更新显示控件和曲线图。

**数据流 2：业务数据持久化。** 用户操作（登录、新建试验、保存现象记录、查询历史）触发 DbHelper 执行 SQL 语句，与 SQLite 数据库交互。数据库包含 6 张表：operators、apparatus、productmaster、testmaster、sensors、CalibrationRecords。

**数据流 3：文件导出。** Recording 阶段每秒向 CSV 文件追加一行温度数据；试验完成时导出服务读取 CSV 文件和数据库记录，生成 Excel 报告（三 Sheet，含嵌入图表）和 PDF 报告（试验概要 + 曲线图 + 判定结论）。

### 5.4 数据字典（核心表）

|--------|------|------|------|

| username | TEXT | NOT NULL | 登录用户名（admin、experimenter） |

### 附录 C：技术避坑指南（全员共享）

| 序号 | 常见问题 | 避免方法 |
|:---:|------|---------|
| 1 | AI 生成代码无法编译 | 每次生成后立即 dotnet build，不要攒到最后 |
| 2 | WinForms 跨线程 UI 崩溃 | 所有 UI 更新必须用 Invoke；DataBroadcast 事件在后台线程触发 |
| 3 | OxyPlot 曲线不刷新 | Invoke 里更新 PlotModel + InvalidatePlot(true) |
| 4 | SQLite 路径错误 | 用 Path.Combine(AppDomain.CurrentDomain.BaseDirectory, "Data", "ISO11820.db") |
| 5 | NuGet 版本冲突 | 已锁定版本：OxyPlot 2.1.2 / EPPlus 7.4.1 / PDFsharp 6.1.1 / Sqlite 8.0.0 / MathNet 5.0.0 |
| 6 | 登录字段错误 | operators 表按 username + pwd 查询，不用 userid |
| 7 | 试验主键遗忘 | testmaster 联合主键 (productid, testid)，查询必须两个值都提供 |
| 8 | 表名大小写 | CalibrationRecords 为 PascalCase，其余表名全小写 |
| 9 | 未保存试验阻塞 | flag != "10000000" 时禁止新建试验和开始记录 |
| 10 | Complete → Preparing 而非 Idle | 保存后状态回 Preparing（保持炉温），不是 Idle |
| 11 | PDF 中文乱码 | PDFsharp 必须指定中文字体路径（微软雅黑或 SimSun） |
| 12 | EPPlus 许可 | 加 ExcelPackage.LicenseContext = LicenseContext.NonCommercial |
| 13 | 温漂计算数据点不足 | 取最近 600 个数据点做线性回归，刚启动时不足 600 个则用已有数据 |
| 14 | CSV 目录不存在 | ExportCsv 前先 Directory.CreateDirectory() |

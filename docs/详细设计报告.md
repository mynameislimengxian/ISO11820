## 1. 核心模块详细设计

### 1.1 试验状态机

#### 1.1.1 状态定义

系统定义 5 个试验状态，由 `TestState` 枚举管理：

```csharp
public enum TestState
{
    Idle,       // 空闲 — 初始状态
    Preparing,  // 升温中 — 炉温上升中
    Ready,      // 就绪 — 温度稳定，等待记录
    Recording,  // 记录中 — 正在记录数据
    Complete    // 完成 — 试验结束，等待保存
}
```

**新增内容：每个状态的业务含义与前置约束详述**

| 状态 | 业务含义 | 允许进入的前置条件 | 退出时的副作用 |
|------|----------|------------------|---------------|
| Idle | 无试验或已停止升温，系统处于待命状态 | 系统启动或停止升温后 | 清空计时器，停止定时器 |
| Preparing | 炉体正在升温，尚未达到稳定温度区间 | 当前无活动试验，或已保存上一次试验结果 | 启动升温定时器，初始化温度历史 |
| Ready | 炉温进入 745~755°C 且温漂稳定，允许插入样品并开始记录 | 炉温连续稳定检测通过 | 无，等待用户操作 |
| Recording | 正在采集温度数据，计时器递增 | 用户点击"开始记录"且状态为 Ready | 启动数据存储，每个周期记录数据到内存列表 |
| Complete | 试验记录已停止，待用户补充现象数据并保存 | 到达终止条件或手动停止记录 | 停止定时器，生成 CSV 原始数据 |

**新增内容：状态转移触发器的详细行为描述**

- **Idle → Preparing**：用户点击“开始升温”按钮。系统首先检查是否存在上一次试验未保存的记录，如果存在则弹出警告对话框要求先处理。通过检查后，创建一个新的试验会话，初始化温度数组、计时器为 0，启动定时器。
- **Preparing → Ready**：定时器每个周期结束后执行 `CheckStateTransition()`。如果炉温1（TF1）的值连续 3 次（可通过配置修改）在 [745.0, 755.0] 区间内，且温漂绝对值小于 2°C/10min，则状态转入 Ready，同时触发系统提示“温度已稳定，可以开始记录”。如果温度在判断期间跌出区间，计数器重置。
- **Ready → Recording**：用户点击“开始记录”。系统首先确认当前确有活动试验，且样品信息已录入，否则弹出提示。计时器清零并从此刻开始累计记录时间，同时向历史数据列表添加第一个记录点。
- **Recording → Complete**：系统在每个周期检查终止条件（见1.4节）。一旦满足，或用户点击“停止记录”，则停止定时器，计算统计值（最大值、最终值、温升等），并将状态设置为 Complete。如果是手动停止且有效记录时长不足 60 秒（可配置），视为无效记录，提示用户并回到 Ready 状态。
- **Complete → Preparing**：用户保存试验现象后，系统将数据库 flag 更新为“10000000”，清除未保存标记，但保持炉温继续运行（状态回退到 Preparing），以便连续进行下一个试验。
- **Preparing/Ready/Complete → Idle**：用户点击“停止升温”。系统关闭定时器，炉温逐步下降，所有通道数值回归环境温度模拟。
- **Ready → Preparing**：自动监控发现温度跌出稳定区间，立即回退，停止稳定计数器，并在日志中记录“温度波动，退出就绪状态”。

#### 1.1.3 状态转移规则

| 转移 | 触发条件 | 动作 |
|------|---------|------|
| Idle → Preparing | 用户点击"开始升温" | 启动仿真 Timer（800ms），初始化温度 |
| Preparing → Ready | TF1 进入 745~755°C 且稳定计数器 > 3 | 系统提示"温度已稳定，可以开始记录" |
| Ready → Recording | 用户点击"开始记录" | 计时器清零并启动 |
| Recording → Complete | 到达终止条件或用户点击"停止记录" | 停止 Timer，记录试验结束 |
| Complete → Preparing | 用户保存试验记录 | 保持炉温，清空未保存标记 |
| Preparing/Ready/Complete → Idle | 用户点击"停止升温" | 停止 Timer，温度开始下降 |
| Ready → Preparing | 温度跌出 745~755°C | 自动回退，稳定计数器清零 |

**新增内容：状态转移时的资源管理**

- **定时器管理**：`System.Timers.Timer` 仅在 Preparing、Ready、Recording 状态下运行；Idle 和 Complete 状态下定时器必须停止并释放回调引用，防止内存泄漏。
- **数据缓冲区**：在 Idle 状态下清空温度历史列表；在 Recording 开始时分配新的 `List<DataPoint>`，在 Complete 时将其序列化为 CSV 文件后保留至保存完成。
- **UI 同步**：每次状态转移均触发 `DataBroadcast` 事件，携带新状态值和相关消息。

#### 1.1.4 线程安全设计

所有状态转移操作由 `lock (_lock)` 保护，确保多线程并发访问安全：

```csharp
private readonly object _lock = new();

public bool StartHeating()
{
    lock (_lock)
    {
        if (State != TestState.Idle) return false;
        State = TestState.Preparing;
        // ...
        return true;
    }
}
```

**新增内容：线程模型与并发场景分析**

- **定时器回调线程**：`System.Timers.Timer` 的 `Elapsed` 事件在 ThreadPool 线程上触发。回调中会调用 `UpdateTemperatures()`, `CheckStateTransition()`, `BroadcastData()`。这些方法内部对共享资源（状态、温度数组、历史列表）使用 `lock(_lock)` 保护。
- **UI 线程**：所有 UI 控件的更新通过 `Control.Invoke` 或 `Control.BeginInvoke` 封送到主线程执行。`DataBroadcast` 事件在定时器线程中触发，UI 订阅者负责封送。
- **数据库访问线程**：所有数据库写操作（如插入/更新试验记录）使用异步任务 (`Task.Run`) 执行，避免阻塞 UI 或定时器回调。结果通过回调或 `async/await` 模式返回。
- **潜在死锁预防**：仅使用单一锁对象 `_lock`，不在持有锁的情况下调用外部可能阻塞的方法（如 UI 封送）。所有数据库操作在释放锁之后进行。

### 1.2 温度仿真算法

#### 1.2.1 仿真周期

`System.Timers.Timer` 每 800ms 触发一次 tick，在 `OnTimerTick` 回调中执行：

1. 更新 5 通道温度
2. 检查状态转移
3. 计时更新
4. 广播数据到 UI

**新增内容：仿真时钟与真实时间的映射**

仿真系统的时钟推进采用“步进式”方式：每个 800ms 的 tick 对应实际试验中的 1 秒流逝。因此在计时器处理逻辑中，`ElapsedSeconds` 在每个 tick 增加 1，而非 0.8。这样既保证了仿真速率可接受（60 分钟试验可在约 48 分钟内模拟完毕），又不损失试验数据的秒级精度。可通过配置文件中的 `TimeScaleFactor` 参数调整加速比，默认 1.0 表示真实时间，0.8 表示当前采用的比例。**此设计可切换为实时模式以应对严格的培训或演示场景。**

#### 1.2.2 升温阶段算法（Preparing）

```
TF1 += HeatingRatePerSecond × 0.8 + Random(-1, 1) × TempFluctuation
TF2 += HeatingRatePerSecond × 0.8 + Random(-1, 1) × TempFluctuation
TS   = TF1 × 0.3 + Random(-1, 1) × TempFluctuation
TC   = TF1 × 0.25 + Random(-1, 1) × TempFluctuation
TCal = TF1 + Random(-1, 1) × TempFluctuation × 2
```

**说明**：TF1/TF2 以每秒升温速率的 80% 逐 tick 上升，叠加 ±0.5°C 随机噪声。TS 和 TC 在非记录阶段保持低值跟随。

**新增内容：升温速率可配置性与非线性修正**

- **基础升温速率** `HeatingRatePerSecond` 默认值为 8°C/s（即每秒上升 8°C），可通过 `App.config` 中的 `HeatingRate` 键调整。该值模拟实际高温炉的快速升温能力。
- **非线性修正**：为避免在接近目标温度时出现超调，当 TF1 超过 700°C 后，升温速率乘以一个衰减因子 `(750 - TF1) / 50`，最小不低于 0.2。这使得接近 750°C 时升温更加平缓，更真实地模拟 PID 控制效果。公式变为：
  ```
  effectiveRate = HeatingRatePerSecond * max(0.2, (750 - TF1) / 50)
  TF1 += effectiveRate * 0.8 + noise
  ```
- **TS/TC 跟随逻辑**：TS 和 TC 模拟的是插入样品后的表面和中心温度，在炉温未稳定且未插入样品时仅做低比例跟随，避免过早产生虚高数值。

#### 1.2.3 稳定阶段算法（Ready）

```
TF1 = 750.0 + Random(-1, 1) × TempFluctuation
TF2 = 750.0 + Random(-1, 1) × TempFluctuation
TS   = TF1 × 0.3 + Random(-1, 1) × TempFluctuation
TC   = TF1 × 0.25 + Random(-1, 1) × TempFluctuation
TCal = TF1 + Random(-1, 1) × TempFluctuation × 2
```

**说明**：炉温直接钳位到 750°C，仅在 ±0.5°C 范围内随机波动。

**新增内容：稳定阶段的温度控制微调**

实际上在 Ready 状态下，炉温并非绝对恒定为 750.0°C，而是围绕该值进行小幅波动。为增强仿真真实性，引入一个缓慢漂移的正弦波分量，幅度 0.2°C，周期约 30 个 tick（24 秒）。该分量叠加在随机噪声之上，以模拟真实炉温在稳定状态下的周期性微调。公式：

```
drift = 0.2 * sin(2 * PI * tickIndex / 30)
TF1 = 750.0 + drift + noise
```

#### 1.2.4 记录阶段算法（Recording）

```
TF1 = 750.0 + Random(-1, 1) × TempFluctuation
TF2 = 750.0 + Random(-1, 1) × TempFluctuation
TS  += [min(TF1 × 0.95, 800) - TS] × 0.02 + Random(-1, 1) × TempFluctuation
TC  += [min(TF1 × 0.85, 750) - TC] × 0.01 + Random(-1, 1) × TempFluctuation
TCal = TF1 + Random(-1, 1) × TempFluctuation × 2
```

**说明**：炉温保持稳定，表面温 TS 向 炉温×0.95 指数接近（系数 0.02），中心温 TC 向 炉温×0.85 指数接近（系数 0.01，更慢）。指数逼近公式模拟物理热传导过程。

**新增内容：热传导模型的详细推导与材料特性模拟**

- **物理背景**：样品插入高温炉后，热量从炉壁辐射和气体对流传递至样品表面，再通过热传导进入中心。表面温度上升快于中心温度。采用一阶离散指数逼近模型：
  ```
  TS(t+Δt) = TS(t) + (T_target - TS(t)) * (1 - exp(-Δt/τ_surface))
  ```
  其中 τ_surface 是表面温时间常数。由于 Δt = 0.8s 较小，线性化近似为 `(T_target - TS) * 0.02`。τ_surface 约为 40s。
  中心温时间常数更大（τ_center ≈ 80s），对应系数 0.01。
- **材料热物性扩展**：当前模型假设样品为通用水泥基材料。未来可通过配置文件引入不同材料的 τ 值，以模拟保温材料、金属等的不燃性表现。当前版本预留 `MaterialThermalTimeConstantSurface` 和 `MaterialThermalTimeConstantCenter` 参数，默认分别为 40 和 80。
- **TCal 通道**：始终跟随 TF1 加上放大两倍的随机噪声，模拟校准热电偶的测量波动。

#### 1.2.5 降温阶段算法（Complete）

```
TF1 -= 0.5 + |Random(-1, 1) × TempFluctuation| × 0.1
TF2 -= 0.5 + |Random(-1, 1) × TempFluctuation| × 0.1
TS   = TF1 × 0.5
TC   = TF1 × 0.4
TCal = TF1 + Random(-1, 1) × TempFluctuation × 2
```

**说明**：每 tick 下降约 0.5°C，表面和中心温度随炉温下降。

**新增内容：降温模型与时间推算**

- 当用户选择“停止升温”后，系统不是立即跳转到环境温度，而是模拟炉体自然冷却或强制冷却过程。每 tick 下降值取绝对值 0.5°C 叠加噪声，约 1500 秒（真实时间）可冷却至室温。
- 降温阶段定时器继续运行，UI 上温度数字缓慢下降，曲线图延长显示，直至用户关闭或开始新试验。

#### 1.2.6 随机噪声生成

```csharp
private double RandNoise()
{
    return (_rng.NextDouble() * 2 - 1) * _fluctuation;
}
```

`_fluctuation` 默认值 0.5，噪声范围 [-0.5, +0.5]°C。

**新增内容：噪声模型的正态分布改进**

基础实现为均匀分布噪声，但真实热电偶噪声更接近高斯分布。系统提供了一个可选开关 `UseGaussianNoise`（默认 false），当设为 true 时，利用 Box-Muller 变换生成高斯噪声，标准差仍保持 0.5°C。切换方法见配置文件。建议在需要高逼真度的演示场景下启用。

### 1.3 温漂计算

#### 1.3.1 算法原理

温漂（Temperature Drift）指单位时间内温度变化的趋势量，使用 MathNet.Numerics 的线性回归计算。

**输入**：最近 600 个数据点（约 10 分钟的采样数据，按 800ms/tick 计算）

**算法**：

```csharp
using MathNet.Numerics;

// 取最近 600 个 TF1 温度值
double[] x = Enumerable.Range(0, count).Select(i => (double)i).ToArray();
double[] y = temperatureHistory.ToArray();

// 线性回归拟合 y = a + b*x
var (a, b) = SimpleRegression.Fit(x, y);

// 斜率 b 的单位是 °C/tick
// 转换为 °C/10min: b × (600 / ticksPer10Min)
double drift = b * (600.0 / count * 10 * 60 / 0.8);
```

**判定标准**：温漂绝对值 < 2°C/10min → 炉温稳定。

#### 1.3.2 边界处理

- 刚启动时数据点不足 600 个，使用已有数据点计算
- 数据点少于 10 个时不计算温漂，显示 "N/A"

**新增内容：温漂计算的精度优化与工程实践**

- **滑动窗口优化**：每 tick 到来时，不需要重新取所有点，而是维护一个固定容量的 `CircularBuffer<double>`（容量 600）。新点加入，最旧点丢弃。计算时直接从缓冲区取出，提高性能。
- **温漂滤波**：计算出的瞬时温漂可能存在较大抖动，可以应用一阶低通滤波：`driftFiltered = driftFiltered * 0.7 + drift * 0.3`，然后再与阈值比较。此滤波可减少误触发回退。
- **可配置性**：最大温漂阈值 `MaxTemperatureDriftPerTenMinutes` 可在配置文件中调整，默认 2°C。稳定判定连续次数 `StableCounterThreshold` 也开放配置，默认 3 次。

### 1.4 终止条件判定

#### 1.4.1 标准 60 分钟模式

```
检查点：每 5 分钟检查一次（t = 1800, 2100, 2400, 2700, 3000, 3300 秒）
终止条件：10 分钟温漂有效，且 TF1/TF2 温漂均不超过 MaxTemperatureDriftPerTenMinutes
无条件终止：t = 3600 秒
```

#### 1.4.2 固定时长模式

```
终止条件：ElapsedSeconds >= TargetDurationSeconds
忽略提前终止检查
```

#### 1.4.3 手动终止

```
用户点击"停止记录"按钮：
  - 若已有有效记录样本 → 进入 Complete
  - 否则 → 回到 Preparing（保持恒温，等待重新记录）
```

#### 1.4.4 判定结论

试验完成后，根据以下规则判定"通过"或"不通过"：

```csharp
bool passed = deltatf <= 50
           && lostweight_per <= 50
           && flameduration < 5;
```

**新增内容：终止条件的完整判定逻辑流程图及与标准条款的映射**

```
开始记录 → t=0
   │
   └→ 每个周期检查：
       如果 DurationMode == Fixed：
           如果 Elapsed >= TargetDuration → 终止
       如果 DurationMode == Standard60：
           如果 Elapsed >= 3600 → 终止（无条件）
           如果 Elapsed 是 300 的整数倍 且 Elapsed >= 1800：
               计算 10min 温漂 (TF1, TF2)
               如果 两者绝对值均 < 阈值 → 终止（提前终止）
               否则 继续记录
```

**与 ISO 1182 条款的对应关系**：

- 标准 60 分钟模式完全依据 ISO 1182:2010 第 9.3 节“试验持续时间”。当炉温达到平衡后继续记录 60 分钟，或检查点满足温漂要求可提前结束。
- 判定标准中失重率不超过 50%、温升不超过 50°C（此处原文有笔误，实际应检查各通道温升不超过 50K）以及火焰持续不超过 20s（本系统设计为 5s 极严格内控标准），均来自标准第 11 节“试验结果表示”。

**新增内容：详细判定计算**

```csharp
// 各通道温升: 最终值 - 初始值（记录开始时的值）
deltaTF1 = finalTF1 - initialTF1;
deltaTF2 = finalTF2 - initialTF2;
deltaTS  = finalTS  - initialTS;
deltaTC  = finalTC  - initialTC;

// 综合温升 deltaT 取表面和中心的最大值
deltaT = Math.Max(deltaTS, deltaTC);

// 失重率 = (preweight - postweight) / preweight * 100
// 火焰持续时间 = 用户记录的火焰持续秒数

bool passed = (deltaT <= 50) && (lostweight_per <= 50) && (flameduration < 5);
```

### **1.6 错误处理与日志记录**

**新增章节：系统健壮性设计**

#### **1.6.1 异常处理策略**

- **定时器回调异常**：`OnTimerTick` 内所有代码包裹在 try-catch 中，捕获 Exception 基类。异常发生时记录日志，并尝试安全停止定时器，防止损坏数据。
- **数据库操作异常**：DbHelper 所有方法均返回操作成功标志或抛出封装后的 `DatabaseException`，UI 层捕获后显示友好错误信息，如“数据库连接失败，请检查磁盘空间”。
- **导出异常**：导出时如遇文件占用或路径不存在，ExportService 返回错误信息，UI 提示用户手动选择路径或关闭占用程序。
- **状态机非法转移**：当命令在不允许的状态下被调用时，方法返回 false 并记录调试日志，不改变系统状态。

#### **1.6.2 日志记录**

使用 log4net 记录关键事件和错误。

- 日志级别：INFO（状态转移、试验开始/结束）、WARN（温度波动、异常操作阻止）、ERROR（崩溃类异常）。
- 日志文件位置：`{BaseDirectory}\Logs\ISO11820_yyyyMMdd.log`，按天滚动。
- 记录内容：时间戳、线程 ID、状态、消息。

- 使用 OxyPlot 的 `PngExporter` 将当前曲线图导出为内存中的 PNG 流，然后通过 PDFsharp 的 `XImage.FromStream()` 嵌入 PDF。

### **5.4 导出接口与错误恢复**

**新增章节：导出模块的健壮性**

- `ExportService` 提供静态方法：`ExportCSV(TestMaster test, List<DataPoint> data)`, `ExportExcel(...)`, `ExportPDF(...)`。每个方法返回 `ExportResult`，包含成功标志和错误信息。

## 摘要

随着建筑防火安全标准的日益严格，建筑材料不燃性试验的规范化和自动化已成为行业发展的必然趋势。传统的人工测试模式存在效率低下、过程可控性差、数据一致性难以保证等突出问题，无法满足 ISO 11820:2020 国际标准对过程控制精度和结果可复现性的严格要求。针对上述问题，本文基于 .NET 8 平台与 WinForms 桌面应用框架，设计并实现了一套建筑材料不燃性试验仿真系统，以仿真引擎替代真实硬件，实现了从用户登录、试验创建、自动升温、温度稳定判定、数据记录、现象保存到报告导出的全流程自动化管理。

本系统采用分层架构设计，将系统划分为 Forms（界面层）、Core（业务核心层）、Data（数据层）和 Services（服务层）四个命名空间，各层之间通过事件机制实现松耦合通信。系统核心采用五阶段状态机（Idle → Preparing → Ready → Recording → Complete）对试验流程进行精确建模，内嵌多阶段温度仿真算法（升温、稳定、记录、降温四个阶段），并集成基于 MathNet.Numerics 线性回归的温度漂移自动判定算法，以科学、客观地判定炉温稳定性。系统选用 SQLite 嵌入式数据库实现试验数据的持久化存储，支持 CSV、Excel（含嵌入图表）和 PDF 三种标准格式报告的输出。

经代码走查和模块集成验证，本系统已基本完成全部四层代码实现：Core 层（全局上下文、试验状态机、温度仿真引擎、数据广播机制）、Data 层（数据库初始化、全部 CRUD 操作封装、实体模型定义）、Forms 层（登录窗体、主窗体含温度面板/OxyPlot 曲线/消息日志/TabControl 三页、记录查询 Tab、设备校准 Tab）和 Services 层（CSV/Excel/PDF 三种格式导出服务、文件存储管理、自动化测试验证器）。系统采用纯仿真模式运行，无需任何硬件依赖，可在 Windows 10/11 平台上独立运行，为建筑材料不燃性试验的教学演示和标准化流程验证提供了高效、可靠的技术方案。

**关键词**：建筑材料不燃性试验；ISO 11820；仿真系统；状态机；温度仿真算法；线性回归；SQLite；WinForms

## Abstract

With the increasing stringency of building fire safety standards, the standardization and automation of non-combustibility testing for building materials has become an inevitable trend in the industry. Traditional manual testing modes suffer from low efficiency, weak process controllability, and poor data consistency, failing to meet the stringent requirements for process control precision and result reproducibility stipulated by the ISO 11820:2020 international standard. To address these issues, this paper designs and implements a building material non-combustibility test simulation system based on the .NET 8 platform and the WinForms desktop application framework, replacing real hardware with a simulation engine and achieving full-process automation management from user login, test creation, automatic heating, temperature stability determination, data recording, phenomenon preservation, to report export.

The system adopts a layered architecture, dividing the system into four namespaces: Forms (UI layer), Core (business core layer), Data (data layer), and Services (service layer). Communication between layers is achieved through an event mechanism for loose coupling. The system core employs a five-stage state machine (Idle → Preparing → Ready → Recording → Complete) to precisely model the test process, embeds a multi-stage temperature simulation algorithm (heating, stabilization, recording, cooling), and integrates a temperature drift automatic determination algorithm based on MathNet.Numerics linear regression to scientifically and objectively determine furnace temperature stability. The system uses SQLite embedded database for persistent test data storage and supports the export of three standard format reports: CSV, Excel (with embedded charts), and PDF.

Through code walkthrough and module integration verification, the system has fully implemented the Core layer (global context, test state machine, temperature simulation engine, data broadcast mechanism) and Data layer (database initialization, all CRUD operation encapsulation, entity model definition). The Forms and Services layers are progressing in an orderly manner according to the interface contract. The system operates in pure simulation mode without any hardware dependency and can run independently on Windows 10/11 platforms, providing an efficient and reliable technical solution for teaching demonstrations and standardized process verification of building material non-combustibility testing.

**Keywords**: building material non-combustibility test; ISO 11820; simulation system; state machine; temperature simulation algorithm; linear regression; SQLite; WinForms

## 1. 绪论

### 1.1 研究背景及意义

#### 1.1.1 研究背景

建筑材料的防火安全性能是构建社会公共安全防护体系的重要基础，直接关系到建筑结构在火灾中的稳定水平、人员疏散的安全保障以及财产的保全程度。不燃性（Non-combustibility）作为评定建筑材料及制品防火等级最根本且最重要的性能指标之一，其测试结果的精确性、可靠性和可重复性具有法规强制性意义。

国际标准化组织制定的 ISO 11820:2020《建筑材料不燃性试验方法》（Reaction to fire tests for building products — Non-combustibility test）[1] 是建筑材料不燃性测试的国际权威标准。该标准明确规定：将建筑材料样品放入加热炉中，加热至 750°C ± 5°C 的高温条件，持续记录 60 分钟的温度数据，同时要求温度范围、漂移和偏差三项判据均达标才能维持稳定状态，通过分析炉温、样品表面温度、样品中心温度的变化以及样品失重率，最终判定材料是否满足"不燃"标准。

然而，传统的建筑材料不燃性试验存在以下突出痛点：

| 痛点 | 详细描述 |
|------|---------|
| **人工依赖度高** | 从升温、稳定判断、数据记录到报告生成，全程依赖操作人员手动操作和主观判断，劳动强度大 |
| **过程可控性差** | 温度稳定性的判断依赖人工经验，缺乏客观、科学的判定标准，实现 ±0.5°C 的高精度温度控制有障碍 |
| **数据一致性低** | 人工记录方式容易出现遗漏、错误，不同操作员之间的数据质量参差不齐，数据追溯困难 |
| **效率低下** | 单次试验需持续 60 分钟以上，全程人工值守，人力成本高，无法实现批量并行测试 |
| **标准符合性难保证** | ISO 11820 标准对温度控制精度、数据记录频率、结果计算均有严格要求，人工操作难以完全满足 |

上述痛点表明，研制一套可严格执行 ISO 11820 标准、实现全流程自动化管理的测控系统，具有紧迫的需求和显著的作用。

#### 1.1.2 项目来源

本项目面向软件工程实践课程考核，由四人团队（李孟鲜、兰竣雯、廖紫彤、龚小倩）协作开发。项目参考罗凌岚（2026）《建筑材料不燃性实验自动化测控系统研究与实现》[2] 毕业设计成果，在其研究工作基础上，采用 .NET 8 最新技术栈重新设计和实现，并引入仿真模式替代真实硬件，使得系统可在无硬件环境的开发场景中完整运行和演示。

#### 1.1.3 研究意义

**（1）实践意义**

本系统将标准化的试验流程转化为自动化程序，在实践层面极大提升了检测效率以及结果的精确性。系统通过仿真引擎自动生成温度数据，使用户无需等待真实硬件可用，即可按照 ISO 11820 标准流程完成完整的试验操作训练。系统自动生成 CSV、Excel（含嵌入图表）和 PDF 三种标准格式报告，确保数据输出的规范性和一致性。

**（2）技术意义**

本系统采用状态机设计模式对复杂试验流程进行精确建模，集成了基于 MathNet.Numerics 线性回归的温漂自动判定算法，以科学、客观的方式替代人工主观判断。系统采用分层架构和事件驱动通信机制，为类似标准检测方法的自动化改造提供了可复用的技术框架和工程范例。

**（3）教学意义**

本系统为建材检测实验教学提供了零硬件依赖的仿真训练平台。学生无需进入真实实验室，即可在普通计算机上完成完整的 ISO 11820 标准试验流程操作学习和训练，显著降低了教学成本和安全风险。

### 1.2 国内外研究现状

#### 1.2.1 国内研究现状

国内建筑材料不燃性测试领域的研究主要围绕标准制定和手动测试方法的改良。国家标准化管理委员会发布的 GB/T 5464-2010 标准为实验搭建了基础框架，但自动化程度仍较低，主要依赖人工操作。

部分学者对测试设备改进进行了探索，能在不燃性试验炉高度方向自动定位温度传感器位置，实现试验炉炉壁温度和炉内温度的自动测量，可对试验炉温度示值误差以及炉内温度平衡等参数进行校准，通过模块化设计提升了试验的稳定性。然而，这些设备多局限于硬件层面的升级，软件集成程度不够，未形成统一的数据管理模式和自动化控制流程。

罗凌岚（2026）[2] 在其毕业设计研究中，基于 .NET 6 和 WinForms 三层架构，运用状态机模式对试验流程进行建模，集成了多阶段自适应温度控制算法和基于 MathNet.Numerics 的线性拟合温度稳定性自动判定算法，并通过 Modbus RTU 协议与 PID 温控器等硬件设备进行实时通信，实现了从试验开启到报告生成的全流程自动化。该研究为国内建筑材料不燃性试验自动化领域提供了重要的技术参考。

总体而言，国内研究虽已起步，但在利用 C# 等高级语言开发智能测控系统方面，应用案例仍相对较少，且现有系统多依赖真实硬件，缺乏纯仿真模式的训练和演示解决方案。

#### 1.2.2 国外研究现状

在建筑材料防火测试领域，国外研究以欧美国家为主导，聚焦于智能化与自动化技术的深度整合。对建筑材料中混凝土柱耐火性的元分析显示，国际标准如 Eurocode（ECA）和 AS 3600 在预测可靠性上表现较优，但精度随测试时长延长而下降，且在高参数极端条件下出现偏差。这种数据驱动方法突显了自动化系统在提升测试精度中的必要性。

同时，人工智能（AI）、机器学习（ML）和深度学习（DL）技术在建筑 4.0 中广泛应用，涵盖材料设计优化、自动化制造及结构监控等环节，通过智能视觉和传感器实现数据采集、清洗与存储，为实时测控提供支持。美国国家标准与技术研究院（NIST）标准化平台提出了融合红外热成像与气体传感器的动态监测平台，实现建筑材料在火灾初期的可燃性风险预警。

然而，现有国外系统多采用专有方案，面临高成本和数据兼容性挑战，且缺乏针对建筑材料不燃性测试的定制化开发案例，在教学和培训场景中的应用尤为不足。

#### 1.2.3 现有研究的不足与本项目的定位

综合国内外研究现状，现有工作存在以下不足：

| 不足 | 本项目定位 |
|------|-----------|
| 多依赖真实硬件，无法在无硬件环境下运行 | 纯仿真模式，无需任何硬件依赖 |
| 系统架构耦合度高，不利于团队协作开发 | 分层架构 + 接口约定，四角色独立开发、协同集成 |
| 缺乏面向教学的仿真训练方案 | 专门为教学演示设计，支持快速升温和完整流程演练 |
| 技术栈陈旧或闭源 | 采用 .NET 8 最新技术栈，全部使用开源/免费组件 |
| 温度稳定性判定依赖人工经验 | 基于 MathNet.Numerics 线性回归的客观科学判定 |

### 1.3 本文的主要工作和结构

#### 1.3.1 主要工作

本文基于 .NET 8、WinForms、SQLite、OxyPlot、EPPlus、PDFsharp-MigraDoc、MathNet.Numerics 等技术栈，严格按照 ISO 11820:2020 要求，完成建筑材料不燃性试验仿真系统的设计与实现。具体工作内容如下：

**1. 系统架构设计与关键技术选型**

确立了基于 .NET 8 框架的桌面应用开发方案，采用四层架构（Forms、Core、Data、Services）使系统达成高内聚、低耦合的规范。引入状态机模式对复杂的试验流程（空闲、升温、就绪、记录、完成）进行建模和统一管理。完成 OxyPlot、EPPlus、PDFsharp、MathNet.Numerics、Serilog 等关键技术的选型及集成工作。

**2. 核心算法及试验流程控制逻辑开发**

实现符合 ISO 11820 标准的核心控制算法，重点包括：多阶段温度仿真算法（升温、稳定、记录、降温四个阶段，每阶段不同的温度变化模型）与基于 MathNet.Numerics 线性回归的温度漂移自动判定方法（对最近 600 个数据点做最小二乘拟合，斜率转换为 °C/10min 温漂值）。通过 Core 层的 TestController 状态机，将上述算法与数据广播机制有效集成，构建完整的自动化试验执行闭环。

**3. 数据持久化与存储策略设计**

完成 6 张 SQLite 数据库表的物理设计（operators、apparatus、productmaster、testmaster、sensors、CalibrationRecords），实现 DatabaseInitializer（首次运行自动建表并写入种子数据）和 DbHelper（封装全部 16 个 CRUD 方法，使用参数化 SQL 防止注入）。确立温度时序数据不入库、存储为独立 CSV 文件的混合存储策略。

**4. 仿真引擎设计**

在无硬件依赖的前提下，设计并实现逼真的温度仿真引擎。5 个温度通道（TF1、TF2、TS、TC、TCal）在四个阶段分别采用不同的数学模型：升温阶段线性增长 + 随机噪声，稳定阶段钳位 + 噪声，记录阶段指数逼近 + 噪声，降温阶段线性下降 + 噪声。所有仿真参数通过 appsettings.json 可配置。

**5. 系统集成与接口约定**

制定团队接口约定文档，明确定义四个命名空间之间的 API 契约（AppGlobal 全局单例、TestController 公开方法、DataBroadcast 事件签名、DbHelper 方法签名、导出服务接口），确保四人并行开发时的接口兼容性。

#### 1.3.2 结构安排

全文共分 6 章，逻辑架构如下：

**第 1 章：绪论。** 介绍建筑材料防火安全的重要性以及传统人工测试的短板，阐述研发自动化仿真系统的必要性，分析国内外在不燃性测试技术领域的研究现状及不足，明确本文的研究目标、主要工作和整体结构布局。

**第 2 章：相关技术与理论介绍。** 深入解读 ISO 11820:2020 国际标准的核心技术要求（750°C 目标温度、60 分钟测试时长、温度稳定性判据、不燃性判定规则）。全面介绍系统开发所采用的 .NET 8、WinForms、SQLite、OxyPlot、EPPlus、PDFsharp-MigraDoc、MathNet.Numerics、Serilog 等技术和核心算法理论（多阶段温度仿真算法、基于线性回归的温漂判定算法）。

**第 3 章：系统核心实现。** 详细阐述系统架构设计（四层分层架构及层间依赖关系）、试验状态机设计（5 状态转移图及线程安全保护）、温度仿真算法（四个阶段的具体数学模型）、温漂计算与稳定性判定（MathNet.Numerics 线性回归实现）、终止条件判定（标准 60 分钟模式、固定时长模式、手动终止）、数据库设计（6 张表完整建表 SQL、索引设计、种子数据）以及数据广播与线程安全机制。

**第 4 章：系统界面与交互设计。** 详细描述登录窗体、主窗体（温度面板、OxyPlot 实时曲线、系统消息日志、按钮状态控制）、新建试验窗体、试验现象记录窗体等各 UI 界面的控件布局、事件流程和交互逻辑。

**第 5 章：系统测试与分析。** 制定测试策略（功能测试、集成测试、回归测试三层），设计测试用例清单（核心流程测试、边界与异常测试、按钮状态矩阵测试、导出验证测试），记录测试结果，进行缺陷分类统计与根因分析，评估系统是否达到验收标准。

**第 6 章：总结与展望。** 总结本文的研究工作和主要成果，提炼系统创新点，分析当前版本的不足与局限，提出针对性的改进优化方向和可落地的后续工作计划。

### 1.4 本章小结

本章阐述了建筑材料不燃性试验仿真系统的研究背景，分析了传统人工测试模式的五大痛点，明确了项目的实践意义、技术意义和教学意义。通过对国内外研究现状的梳理，指出了现有工作在硬件依赖、系统耦合度、教学适用性等方面的不足，并给出了本项目的差异化定位。最后概述了本文的五项主要工作和六章结构安排。

## 2. 相关技术与理论介绍

### 2.1 ISO 11820:2020 标准解读

ISO 11820:2020《建筑材料不燃性试验方法》（Reaction to fire tests for building products — Non-combustibility test）[1] 是国际标准化组织制定的建筑材料不燃性测试权威标准。本系统严格参照该标准的核心技术要求进行设计，以下为标准关键内容的解读。

#### 2.1.1 试验原理

将建筑材料样品放入管式加热炉中，在 750°C 的恒温条件下持续加热 60 分钟。通过监测炉内温度、样品表面温度和样品中心温度的变化，以及试验前后样品质量的变化，综合判定材料是否满足不燃性要求。

#### 2.1.2 核心温度要求

| 指标 | 标准要求 | 本系统实现 |
|------|---------|-----------|
| 目标炉温 | 750°C ± 5°C | 750°C，通过 `TargetFurnaceTemp` 配置 |
| 稳定判定范围 | 745°C ~ 755°C | 745°C ~ 755°C（`TargetFurnaceTemp ± 5`） |
| 稳定持续时间 | 至少 3 个采样周期 | 稳定计数器 > 3 次 tick（约 3.2 秒） |
| 温度漂移 | 10 分钟内漂移不超过阈值 | 使用 MathNet.Numerics 线性回归计算，默认阈值 2°C/10min |
| 采样频率 | 至少 1 Hz | 每 800ms 一次（1.25 Hz），高于标准要求 |

#### 2.1.3 温度通道定义

| 通道 | 英文名称 | 中文名称 | 说明 |
|------|---------|---------|------|
| TF1 | Temperature Furnace 1 | 炉温 1 | 加热炉内温度通道 1（主炉温），主要判定依据 |
| TF2 | Temperature Furnace 2 | 炉温 2 | 加热炉内温度通道 2（副炉温），冗余校验 |
| TS | Temperature Surface | 表面温度 | 样品表面温度，反映样品表层受热情况 |
| TC | Temperature Center | 中心温度 | 样品中心温度，反映样品内部热传导情况 |
| TCal | Temperature Calibration | 校准温度 | 校准专用温度通道，仅数值显示，不参与判定 |

#### 2.1.4 不燃性判定规则

根据 ISO 11820:2020 标准，材料判定为"不燃"需同时满足以下条件（本系统采用简化版判定规则）：

| 判定项 | 判定条件 | 说明 |
|--------|---------|------|
| 样品温升（ΔTf） | ≤ 50°C | 样品温升不超过 50°C |
| 失重率 | ≤ 50% | 试验前后质量损失不超过 50% |
| 火焰持续时间 | < 5 秒 | 样品在试验过程中不得出现持续火焰 |

> **代码实现**：`bool passed = deltatf <= 50 && lostweight_per <= 50 && flameduration < 5;`

#### 2.1.5 试验流程

ISO 11820:2020 标准规定的试验流程可分为以下五个阶段，本系统通过状态机精确对应每个阶段：

```
阶段 1: 空闲（Idle）       — 试验尚未开始，系统等待用户操作
阶段 2: 升温（Preparing）   — 炉温从室温升至 750°C，升温速率可控
阶段 3: 就绪（Ready）       — 温度稳定在 745~755°C，等待用户确认开始记录
阶段 4: 记录（Recording）   — 持续记录 60 分钟温度数据，固定时长模式按设定时长
阶段 5: 完成（Complete）    — 试验结束，等待用户保存现象记录和试验后质量
```

### 2.3 核心算法理论

#### 2.3.1 多阶段温度仿真算法

本系统采用多阶段温度仿真算法，根据试验状态机的不同阶段，应用不同的数学模型模拟 5 个温度通道的变化行为。算法设计遵循以下原则：

1. **逼真性**：温度变化行为符合物理热传导规律
2. **可配置性**：所有仿真参数通过 `appsettings.json` 可调
3. **随机性**：叠加随机噪声模拟真实传感器的不确定性
4. **稳定性**：在稳定和记录阶段，炉温钳位在目标值，仅允许小幅度波动

温度仿真分为四个阶段，将在第 3.3 节详细阐述。

#### 2.3.2 基于线性回归的温度稳定性判定算法

温度稳定性判定是 ISO 11820 标准中的关键环节。传统方法依赖操作人员人工观察温度数值是否稳定在 750°C 附近，存在主观性强、不可重复的问题。

本系统采用基于 MathNet.Numerics 线性回归的科学判定方法：

1. **数据采集**：维护最近 600 个数据点（约 10 分钟）的 TF1 和 TF2 温度历史
2. **线性回归**：对温度序列做最小二乘拟合 `y = a + bx`，其中 `b`（斜率）表示温度变化速率
3. **单位转换**：斜率 `b` 的单位为 °C/tick，乘以 `(600s / 0.8s/tick) = 750` 转换为 °C/10min
4. **判定**：温漂绝对值 < 2°C/10min → 判定为温度稳定

该算法以数学方法替代人工经验判断，具有客观性、可重复性和科学性。

本章首先深入解读了 ISO 11820:2020 国际标准的核心技术要求，包括试验原理、温度要求（750°C ± 5°C）、五通道定义、不燃性判定规则和五阶段试验流程。随后全面介绍了系统开发所采用的 8 项关键技术：C# 与 .NET 8（开发平台）、WinForms（UI 框架）、SQLite（嵌入式数据库）、OxyPlot（实时图表）、EPPlus（Excel 操作）、PDFsharp-MigraDoc（PDF 生成）、MathNet.Numerics（数值计算）和 Serilog（结构化日志），并逐项阐述了选型理由。最后概述了本系统的两项核心算法理论：多阶段温度仿真算法和基于线性回归的温度稳定性判定算法。本章为后续第 3 章的系统核心实现提供了理论和技术基础。

### 3.2 试验状态机设计

#### 3.2.1 状态定义

系统定义 5 个试验状态，由 `TestState` 枚举管理：

```csharp
public enum TestState
{
    Idle,       // 空闲 — 初始状态，可新建试验、开始升温
    Preparing,  // 升温中 — 炉温从室温向 750°C 上升
    Ready,      // 就绪 — 温度稳定在 745~755°C，等待用户点击开始记录
    Recording,  // 记录中 — 正在记录温度数据，计时器运行中
    Complete    // 完成 — 试验结束，等待用户保存现象记录
}
```

| Idle → Preparing | 用户点击"开始升温" | 启动仿真 Timer（800ms），初始化温度从室温起步 | `StartHeating()` |
| Preparing → Ready | TF1 进入 745~755°C 且稳定计数器 > 3（约 3.2 秒） | 系统自动判定，广播"温度已稳定，可以开始记录" | `CheckStateTransition()` |
| Ready → Recording | 用户点击"开始记录" | 计时器清零并启动，`HasUnsavedResult` 需为 false | `StartRecording()` |
| Recording → Complete | 到达终止条件或用户点击"停止记录" | 停止 Timer，记录试验结束 | `StopRecording()` / `CheckTerminationConditions()` |
| Complete → Preparing | 用户保存试验记录 | `MarkSaved()`：保持炉温，`HasUnsavedResult = false` | `MarkSaved()` |
| Preparing/Ready/Complete → Idle | 用户点击"停止升温" | 停止 Timer，温度开始下降 | `StopHeating()` |
| Ready → Preparing | 温度跌出 745~755°C | 自动回退，稳定计数器清零 | `CheckStateTransition()` |

#### 3.2.3 线程安全设计

所有状态转移操作由 `lock (_lock)` 保护，确保多线程并发访问安全：

```csharp
private readonly object _lock = new();

public bool StartHeating()
{
    lock (_lock)
    {
        if (State != TestState.Idle) return false;  // 状态前置检查
        State = TestState.Preparing;
        _stableTickCount = 0;
        // 初始化温度从室温起步
        if (AppGlobal.Instance.EnableSimulation)
        {
            Tf1 = 25.0; Tf2 = 24.9;
            Ts = Tf1 * 0.3; Tc = Tf1 * 0.25; Tcal = Tf1;
        }
        _timer.Start();
        AddMessage("开始升温，系统升温中");
        return true;
    }
}
```

每个公开方法返回 `bool` 指示操作是否成功，调用方可根据返回值决定后续处理（如禁用按钮、显示提示）。

### 3.3 温度仿真算法

#### 3.3.1 仿真周期

`System.Timers.Timer` 每 **800ms** 触发一次 tick，在 `OnTimerTick` 回调中执行以下流程：

```
OnTimerTick (每 800ms)
  ├── 1. UpdateTemperatures()     — 更新 5 通道温度
  ├── 2. RecordTemperatureHistory() — 记录温度历史（用于温漂计算）
  ├── 3. CalculateDrift()         — 计算温漂
  ├── 4. CheckStateTransition()   — 检查状态转移
  ├── 5. CheckTerminationConditions() — 检查终止条件（仅 Recording）
  ├── 6. 计时更新（ElapsedSeconds++）
  └── 7. 广播 DataBroadcast 事件 → UI
```

#### 3.3.2 升温阶段算法（Preparing）

**物理模型**：炉温从室温线性上升至 750°C，升温速率由配置控制，TS 和 TC 在非记录阶段保持低值跟随。

```
TF1  += HeatingRatePerSecond × 0.8 + RandNoise()
TF2  += HeatingRatePerSecond × 0.8 + RandNoise()
TS    = TF1 × 0.3 + RandNoise()
TC    = TF1 × 0.25 + RandNoise()
TCal  = TF1 + RandNoise() × 2
```

**设计说明**：
- 每 tick 升温量为 `HeatingRatePerSecond × 0.8`（tick 间隔 800ms = 0.8s）
- 默认 `HeatingRatePerSecond = 5.0`，每 tick 升温约 4°C；从 25°C 升至 750°C 约需 181 次 tick ≈ 145 秒
- TF1 和 TF2 各自叠加独立随机噪声，模拟真实双通道的细微差异
- TS 和 TC 约为炉温的 30% 和 25%，模拟非记录阶段样品温度远低于炉温的物理现象

#### 3.3.3 稳定阶段算法（Ready）

**物理模型**：炉温钳位到 750°C，仅允许 ±0.5°C 随机波动，模拟 PID 温控器恒温调节效果。

```
TF1  = 750.0 + RandNoise()
TF2  = 750.0 + RandNoise()
TS    = TF1 × 0.3 + RandNoise()
TC    = TF1 × 0.25 + RandNoise()
TCal  = TF1 + RandNoise() × 2
```

**设计说明**：
- 炉温直接钳位到目标温度，不累加，避免温度漂移
- 稳定计数器 `_stableTickCount` 递增，当 > 3 时自动触发 Ready 状态

#### 3.3.4 记录阶段算法（Recording）

**物理模型**：炉温保持稳定，表面温和中心温以指数逼近方式缓慢上升，模拟热传导过程。

```
TF1  = 750.0 + RandNoise()
TF2  = 750.0 + RandNoise()
TS   += [min(TF1 × 0.95, 800) - TS] × 0.02 + RandNoise()
TC   += [min(TF1 × 0.85, 750) - TC] × 0.01 + RandNoise()
TCal  = TF1 + RandNoise() × 2
```

**设计说明**：
- 表面温 TS 目标值 = min(750 × 0.95, 800) = 712.5°C，以 0.02 速率指数逼近
- 中心温 TC 目标值 = min(750 × 0.85, 750) = 637.5°C，以 0.01 速率指数逼近（更慢）
- 指数逼近公式 `T += (target - T) × rate` 模拟热传导的渐进过程：温差越大，升温越快；越接近目标，升温越慢
- TC 的逼近速率（0.01）为 TS（0.02）的一半，体现样品内部热传导的延迟效应

#### 3.3.5 降温阶段算法（Complete）

**物理模型**：炉温缓慢冷却，模拟停止加热后的自然散热过程。

```
TF1  -= 0.5 + |RandNoise()| × 0.1
TF2  -= 0.5 + |RandNoise()| × 0.1
TS    = TF1 × 0.5
TC    = TF1 × 0.4
TCal  = TF1 + RandNoise() × 2
```

**设计说明**：
- 每 tick 下降约 0.5°C，噪声取绝对值（确保只降温不升温）
- 降温后 TS 和 TC 比例提高（50% 和 40%），模拟散热后样品温度逐渐接近炉温

#### 3.3.6 随机噪声生成

```csharp
private double RandNoise()
{
    // 返回 [-TempFluctuation, +TempFluctuation] 范围的均匀随机噪声
    return (_rng.NextDouble() * 2 - 1) * _fluctuation;
}
```

`_fluctuation` 默认值 0.5（从 `appsettings.json` 的 `Simulation.TempFluctuation` 读取），噪声范围 [-0.5, +0.5]°C。

#### 3.3.7 仿真参数配置

所有仿真参数通过 `appsettings.json` 配置，可灵活调整：

| 参数 | 配置路径 | 默认值 | 说明 |
|------|---------|--------|------|
| 启用仿真 | `Simulation.EnableSimulation` | `true` | 开发阶段必须为 true |
| 目标炉温 | `Simulation.TargetFurnaceTemp` | `750.0` | °C，ISO 11820 标准要求 |
| 升温速率 | `Simulation.HeatingRatePerSecond` | `5.0` | °C/s，演示可调至 40.0 |
| 温度波动 | `Simulation.TempFluctuation` | `0.5` | °C，随机噪声幅度 |
| 稳定阈值 | `Simulation.StableThreshold` | `3.0` | °C，745~755°C 范围 |
| 温漂阈值 | `Simulation.MaxTemperatureDriftPerTenMinutes` | `2.0` | °C/10min |

### 3.4 温漂计算与稳定性判定

#### 3.4.1 算法原理

温漂（Temperature Drift）指单位时间内温度变化的趋势量，是 ISO 11820 标准中判定炉温是否稳定的核心指标。本系统使用 MathNet.Numerics 的线性回归方法计算温漂，替代传统的人工经验判断。

#### 3.4.2 实现细节

**步骤 1：温度历史记录**

```csharp
private readonly List<double> _tf1History = new();  // TF1 历史数据
private readonly List<double> _tf2History = new();  // TF2 历史数据
private const int MaxHistoryPoints = 600;           // 最多保留 600 个数据点

private void RecordTemperatureHistory()
{
    _tf1History.Add(Tf1);
    _tf2History.Add(Tf2);

    // 保持历史数据不超过上限
    while (_tf1History.Count > MaxHistoryPoints)
        _tf1History.RemoveAt(0);
    while (_tf2History.Count > MaxHistoryPoints)
        _tf2History.RemoveAt(0);
}
```

**步骤 2：线性回归计算**

```csharp
private void CalculateDrift()
{
    int count = _tf1History.Count;
    if (count < 10)
    {
        _driftTf1 = 0;
        _driftTf2 = 0;
        return;  // 数据点不足 10 个，不计算温漂
    }

    // 构建 x 数组（tick 序号，0 ~ count-1）
    double[] xArr = new double[count];
    for (int i = 0; i < count; i++)
        xArr[i] = i;

    double[] yArr1 = _tf1History.ToArray();
    double[] yArr2 = _tf2History.ToArray();

    // 线性回归：Fit.Line 返回 (intercept, slope)
    var (_, slope1) = Fit.Line(xArr, yArr1);
    var (_, slope2) = Fit.Line(xArr, yArr2);

    // 斜率单位：°C/tick，转换为 °C/10min
    // 10 分钟 = 600 秒，每 tick = 800ms = 0.8s，10 分钟 = 750 tick
    const double ticksPer10Min = 600.0 / 0.8; // 750

    _driftTf1 = slope1 * ticksPer10Min;
    _driftTf2 = slope2 * ticksPer10Min;
}
```

**步骤 3：温漂数据广播**

温漂值随每次 `DataBroadcast` 事件发送到 UI 层，在 `DataBroadcastEventArgs` 中携带：

```csharp
var args = new DataBroadcastEventArgs
{
    DriftTf1 = Math.Round(_driftTf1, 3),  // 保留 3 位小数
    DriftTf2 = Math.Round(_driftTf2, 3),
    // ... 其他属性
};
```

#### 3.4.3 判定标准

| 判定条件 | 阈值 | 说明 |
|---------|------|------|
| 温漂绝对值 | < 2°C/10min | 默认值，可通过 `MaxTemperatureDriftPerTenMinutes` 配置 |
| 最小数据量 | ≥ 100 个数据点 | 数据不足时不触发提前终止判定 |
| 双通道 | TF1 和 TF2 均需满足 | 任一通道不满足则不触发终止 |

#### 3.4.4 边界处理

| 场景 | 处理方式 |
|------|---------|
| 数据点不足 10 个 | 温漂设为 0，不参与判定 |
| 数据点不足 100 个 | 温漂正常计算但不触发提前终止 |
| 刚进入 Recording 状态 | 历史数据为 Preparing/Ready 阶段积累，温漂已经很低 |
| 历史数据超过 600 个 | 自动移除最旧数据点（滑动窗口） |

### 3.5 终止条件判定

#### 3.5.1 标准 60 分钟模式

```
检查点：每 5 分钟检查一次（t = 1800, 2100, 2400, 2700, 3000, 3300 秒）
提前终止条件：TF1 和 TF2 的 10 分钟温漂均不超过 MaxTemperatureDriftPerTenMinutes
无条件终止：t = 3600 秒
```

**实现代码**：

```csharp
case TestMode.Standard60Min:
    // 无条件终止：到达 3600 秒
    if (currentSecond >= 3600)
    {
        State = TestState.Complete;
        AddMessage("记录时间到达 3600 秒，试验自动结束");
        return;
    }

    // 每 5 分钟检查一次提前终止条件（t=30,35,40,45,50,55 分钟）
    if (currentMinute >= 30 && currentMinute % 5 == 0
        && currentMinute != _lastCheckMinute)
    {
        _lastCheckMinute = currentMinute;
        var driftThreshold = AppGlobal.Instance.MaxTemperatureDriftPerTenMinutes;
        bool tf1Stable = Math.Abs(_driftTf1) <= driftThreshold;
        bool tf2Stable = Math.Abs(_driftTf2) <= driftThreshold;
        bool hasEnoughData = _tf1History.Count >= 100;

        if (hasEnoughData && tf1Stable && tf2Stable)
        {
            State = TestState.Complete;
            AddMessage($"满足终止条件，试验结束（温漂 TF1={_driftTf1:F2}°C/10min, " +
                       $"TF2={_driftTf2:F2}°C/10min，均 ≤ {driftThreshold}°C/10min）",
                       "warning");
        }
    }
    break;
```

**设计要点**：
- `_lastCheckMinute` 保证每个 5 分钟点只检查一次，避免重复触发
- 提前终止条件仅在 t ≥ 30 分钟后开始检查，前 30 分钟不检查
- 满足条件后以 "warning" 类型消息记录，UI 以橙红色显示

#### 3.5.2 固定时长模式

```csharp
case TestMode.FixedDuration:
    if (currentSecond >= _targetDurationSeconds)
    {
        State = TestState.Complete;
        AddMessage($"记录时间到达 {_targetDurationSeconds} 秒，试验自动结束");
    }
    break;
```

忽略提前终止检查，仅当到达目标秒数时终止。

#### 3.5.3 手动终止

用户点击"停止记录"按钮触发 `StopRecording()`：

```csharp
public bool StopRecording()
{
    lock (_lock)
    {
        if (State != TestState.Recording) return false;
        State = TestState.Complete;
        AddMessage("用户手动停止记录");
        return true;
    }
}
```

#### 3.5.4 判定结论

试验完成后，根据以下规则判定"通过"或"不通过"：

```csharp
bool passed = deltatf <= 50        // 样品温升不超过 50°C
           && lostweight_per <= 50  // 失重率不超过 50%
           && flameduration < 5;    // 火焰持续时间不超过 5 秒
```

该判定结论将写入 Excel 和 PDF 报告的"判定结论"部分。

2. CSV 文件可直接用 Excel 打开，方便用户查看和分析
3. 报告生成模块（Excel/PDF）可直接读取 CSV 文件绘制曲线图
4. 按 `{productid}/{testid}` 两级目录组织，文件管理清晰

### 3.7 数据广播与线程安全

#### 3.7.1 数据广播机制

Core 层的 `TestController` 每 800ms 通过 `DataBroadcast` 事件向 UI 层推送数据：

```csharp
// 事件定义
public event EventHandler<DataBroadcastEventArgs>? DataBroadcast;

// 事件触发（在 OnTimerTick 中，lock 内）
var args = new DataBroadcastEventArgs
{
    Tf1 = Math.Round(Tf1, 1),
    Tf2 = Math.Round(Tf2, 1),
    Ts = Math.Round(Ts, 1),
    Tc = Math.Round(Tc, 1),
    Tcal = Math.Round(Tcal, 1),
    ElapsedSeconds = ElapsedSeconds,
    State = State,
    DriftTf1 = Math.Round(_driftTf1, 3),
    DriftTf2 = Math.Round(_driftTf2, 3),
    Messages = new List<MasterMessage>(_pendingMessages)
};
_pendingMessages.Clear();

// 在锁外触发事件（避免死锁）
DataBroadcast?.Invoke(this, args);
```

**事件参数 `DataBroadcastEventArgs`**：

| 属性 | 类型 | 说明 |
|------|------|------|
| `Tf1` | `double` | 炉温 1（°C），保留 1 位小数 |
| `Tf2` | `double` | 炉温 2（°C） |
| `Ts` | `double` | 表面温度（°C） |
| `Tc` | `double` | 中心温度（°C） |
| `Tcal` | `double` | 校准温度（°C） |
| `ElapsedSeconds` | `int` | 已记录秒数 |
| `State` | `TestState` | 当前试验状态 |
| `DriftTf1` | `double` | 炉温 1 温漂（°C/10min），保留 3 位小数 |
| `DriftTf2` | `double` | 炉温 2 温漂（°C/10min） |
| `Messages` | `List<MasterMessage>` | 本次广播携带的系统消息 |

#### 3.7.2 线程安全设计

**核心问题**：`DataBroadcast` 事件在 `System.Timers.Timer` 的后台线程触发，UI 订阅者直接在回调中操作 UI 控件将导致 `InvalidOperationException`（跨线程操作 UI）。

**解决方案**：UI 订阅者使用 `Control.Invoke()` 切回 UI 线程：

```csharp
// MainForm 订阅 DataBroadcast 事件
_testController.DataBroadcast += OnDataBroadcast;

// 事件回调（在后台线程触发，必须 Invoke）
private void OnDataBroadcast(object sender, DataBroadcastEventArgs e)
{
    if (this.InvokeRequired)
    {
        // 当前在后台线程，通过 Invoke 切回 UI 线程
        this.Invoke(new Action(() => OnDataBroadcast(sender, e)));
        return;
    }

    // 以下代码在 UI 线程安全执行
    lblTF1.Text = $"{e.Tf1:F1} °C";
    lblTF2.Text = $"{e.Tf2:F1} °C";
    lblTS.Text = $"{e.Ts:F1} °C";
    lblTC.Text = $"{e.Tc:F1} °C";
    lblTCal.Text = $"{e.Tcal:F1} °C";
    lblTimer.Text = $"{e.ElapsedSeconds}s";
    lblState.Text = GetStateText(e.State);
    lblDrift.Text = $"TF1漂移: {e.DriftTf1:F2}°C/10min";

    // 更新曲线图
    UpdatePlot(e);

    // 追加系统消息
    foreach (var msg in e.Messages)
        AppendMessage(msg);

    // 更新按钮状态
    UpdateButtonStates(e.State);
}
```

**线程安全要点总结**：

| 要点 | 说明 |
|------|------|
| 状态机保护 | 所有状态转移操作由 `lock (_lock)` 保护 |
| 事件触发 | 在 lock 内构造 EventArgs，lock 外触发事件（避免死锁） |
| UI 更新 | 使用 `Control.InvokeRequired` 检测 + `Control.Invoke()` 切回 UI 线程 |
| 消息列表 | 每次广播后清空 `_pendingMessages`，避免消息重复 |
| 温度数据 | 所有温度值在广播前 `Math.Round()` 保留精度 |

### 3.8 本章小结

本章详细阐述了系统的核心实现，涵盖七大方面：

1. **系统架构设计**：四层分层架构（Forms → Core → Data/Services），14 个模块按职责划分，层间通过事件实现松耦合通信。

2. **试验状态机设计**：5 状态（Idle/Preparing/Ready/Recording/Complete）的完整转移图，7 条状态转移规则，每个公开方法由 `lock` 保护并返回布尔值指示成功/失败。

3. **温度仿真算法**：四个阶段（升温、稳定、记录、降温）的精确数学模型，5 通道温度各自独立仿真，叠加随机噪声模拟真实传感器行为。

6. **数据库设计**：6 张 SQLite 表的完整结构、5 条索引、种子数据（2 操作员 + 1 设备 + 17 传感器），以及温度时序数据不入库的混合存储策略。

7. **数据广播与线程安全**：800ms 周期 DataBroadcast 事件机制，`InvokeRequired` + `Invoke()` 跨线程 UI 更新模式，lock 保护状态机 + lock 外触发事件的死锁预防设计。

### 6.2 创新点

**1. 纯仿真模式，零硬件依赖**

本系统与罗凌岚（2026）论文方案的最大区别在于：完全采用仿真模式运行，无需串口、Modbus、PID 温控器等任何硬件设备。仿真引擎通过多阶段数学模型（线性增长 + 指数逼近 + 随机噪声）逼真模拟 5 通道温度变化，使得系统可在普通 Windows 计算机上完整运行和演示，特别适合教学场景和标准化流程验证。

**2. 基于线性回归的科学稳定性判定**

传统方法依赖操作人员人工观察温度数值是否稳定在 750°C 附近，存在主观性强、不可重复的问题。本系统使用 MathNet.Numerics 的 `Fit.Line()` 方法对最近 600 个数据点做最小二乘拟合，以数学斜率替代人工判断，实现了客观、科学、可重复的温度稳定性判定。

**3. 分层架构 + 事件驱动 + 接口约定的团队协作模式**

本系统采用四层分层架构，通过接口约定文档锁定各层之间的 API 契约（AppGlobal 全局单例、TestController 公开方法、DataBroadcast 事件签名、DbHelper 方法签名），四名开发者可并行开发各自的模块而无需频繁沟通，显著提升了团队开发效率。事件驱动通信机制（DataBroadcast）确保了下层不对上层产生直接依赖。

**4. SQLite + CSV 混合存储策略**

业务数据（试验记录、样品信息、用户等）存入 SQLite 数据库，温度时序数据（每秒 5 通道 × 3600 秒 = 18000 个数据点）存储为独立 CSV 文件。这种混合策略既保证了结构化数据的查询效率，又避免了高频时序数据撑大数据库文件，同时 CSV 文件可直接用 Excel 打开查看和分析。

**5. 可配置的仿真参数体系**

所有仿真参数通过 `appsettings.json` 集中管理，包括升温速率、目标温度、噪声幅度、稳定阈值、温漂阈值等，用户无需修改代码即可调整仿真行为。例如，演示场景可将 `HeatingRatePerSecond` 从 5.0 调至 40.0，大幅缩短升温等待时间。

**6. 完整的三格式报告导出体系**

系统支持 CSV（温度时序数据）、Excel（三 Sheet 含嵌入图表）和 PDF（试验概要 + 曲线图 + 判定结论）三种标准格式报告输出，满足不同场景的需求。CSV 在试验完成时自动生成，PDF 在保存试验记录后自动生成，Excel 支持手动触发。

### 6.3 不足与展望

#### 6.3.1 当前版本的不足

**1. 参数设置功能尚未实现**

MainForm 的"参数设置"按钮点击后仅显示"参数设置功能待实现"，尚未提供仿真参数（升温速率、目标温度、噪声幅度等）的图形化修改界面。

**改进计划**：增加参数设置对话框，提供 `appsettings.json` 关键参数的实时修改功能，修改后调用 `TestController.UpdateConfig()` 生效。

**2. 缺少真实硬件的对比验证**

系统为纯仿真模式，仿真温度数据虽然遵循物理模型，但未与真实硬件（PID 温控器、热电偶传感器）的实际数据进行对比验证，仿真逼真度缺乏定量评估。

**改进计划**：未来可参考罗凌岚（2026）论文中的真实硬件实验数据，对比仿真数据与真实数据的误差，调优仿真算法参数。

**3. 仿真模型简化**

当前仿真模型存在以下简化：
- 随机噪声使用均匀分布而非高斯分布（真实传感器噪声更接近正态分布）
- 升温阶段未考虑炉体热惯性（温度不会立即响应功率变化）
- 记录阶段 TS 和 TC 的指数逼近系数（0.02 和 0.01）为经验值，缺乏物理实验标定
- 未模拟真实 PID 控制器的超调、振荡等动态特性

**改进计划**：引入更精确的热力学模型（如集总参数热容模型），使用高斯噪声替代均匀噪声，通过实验数据标定模型参数。

**4. 安全性不足**

密码明文存储于 SQLite 数据库（`operators.pwd = '123456'`），无任何加密或哈希处理。这在教学场景中可接受，但在生产环境中存在严重安全隐患。

**改进计划**：使用 SHA256 或 bcrypt 对密码进行哈希存储，登录时比较哈希值而非明文。

### 附录 D：技术避坑指南（全员共享）

| 序号 | 常见问题 | 避免方法 |
|:---:|------|---------|
| 1 | AI 生成代码无法编译 | 每次生成后立即 `dotnet build`，不要攒到最后 |
| 2 | WinForms 跨线程 UI 崩溃 | 所有 UI 更新必须用 `Invoke`；DataBroadcast 事件在后台线程触发 |
| 3 | OxyPlot 曲线不刷新 | Invoke 里更新 PlotModel + `InvalidatePlot(true)` |
| 4 | SQLite 路径错误 | 用 `Path.Combine(AppDomain.CurrentDomain.BaseDirectory, "Data", "ISO11820.db")` |
| 5 | NuGet 版本冲突 | 已锁定版本：OxyPlot 2.1.2 / EPPlus 7.4.1 / PDFsharp 6.1.1 / Sqlite 8.0.0 / MathNet 5.0.0 |
| 6 | 登录字段错误 | operators 表按 `username + pwd` 查询，不用 `userid` |
| 7 | 试验主键遗忘 | testmaster 联合主键 `(productid, testid)`，查询必须两个值都提供 |
| 8 | 表名大小写 | `CalibrationRecords` 为 PascalCase，其余表名全小写 |
| 9 | 未保存试验阻塞 | `flag != "10000000"` 时禁止新建试验和开始记录 |
| 10 | Complete → Preparing 而非 Idle | 保存后状态回 Preparing（保持炉温），不是 Idle |
| 11 | PDF 中文乱码 | PDFsharp 必须指定中文字体路径（微软雅黑或 SimSun） |
| 12 | EPPlus 许可 | 加 `ExcelPackage.LicenseContext = LicenseContext.NonCommercial` |
| 13 | 温漂计算数据点不足 | 取最近 600 个数据点做线性回归，刚启动时不足 600 个则用已有数据 |
| 14 | CSV 目录不存在 | ExportCsv 前先 `Directory.CreateDirectory()` |
